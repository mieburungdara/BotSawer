# Plan: Enhance Media Upload Handling with Detailed Metadata and Improved User Feedback

## Goal
Enhance the bot's media upload functionality to store comprehensive metadata about uploaded media (including user info, album detection, timestamps), forward enriched information to the backup channel for admin oversight, and provide users with detailed queue information and cancellation options.

## Current State Analysis
- Bot currently stores basic media info in `media_files` table: creator_id, telegram_file_id, file_unique_id, file_type, caption, timestamps, status
- Media forwarding to backup channel includes only basic caption
- User confirmation message is basic: "Media diterima dan masuk antrian posting! ID Media: #X Jenis: Y Status: Dalam antrian"
- No album support or detailed queue timing
- No cancellation mechanism

## Proposed Changes

### 1. Database Schema Updates
**File:** `migrations/add_media_metadata_migration.sql`
- Add `bot_id` BIGINT UNSIGNED NULL (which bot received the media - for multi-bot support)
- Add `user_id` BIGINT UNSIGNED NOT NULL (who actually posted the media)
- Add `media_group_id` VARCHAR(255) NULL (Telegram's media group identifier for albums - null for single media, not null for album media)
- Add FOREIGN KEY constraints for bot_id and user_id
- Add composite index on (bot_id, media_group_id) for efficient album queries

### 2. Enhanced Media Processing Logic
**File:** `src/Bot.php` - `handleMediaUpload()` method
- Detect album uploads by checking `media_group_id` in Telegram message
- Store additional metadata in database
- Forward media to backup channel with enriched caption including:
  - Media ID and type
  - Creator info
  - Upload timestamp
  - Album info (if media_group_id is present)
  - File metadata (size, duration if video)

### 3. Improved User Feedback System
**File:** `src/Bot.php` - `handleMediaUpload()` method
- Calculate queue position: count scheduled media with earlier scheduled_at
- Calculate estimated posting time: (queue_position - 1) * 1 minute from now
- Send detailed confirmation message with:
  - Queue number
  - Estimated time to post
  - Cancel button (inline keyboard)

### 4. Cancel Media Functionality
**File:** `src/Bot.php` - Add new method `handleCancelMediaCallback()`
- Add callback handler for 'cancel_media_X' pattern
- Allow cancellation for any non-posted status ('queued', 'scheduled', 'posting')
- Update media status to 'cancelled'
- Notify user of cancellation

**File:** `schedule.php` - Add safety check
- Double-check media status before posting to prevent cancelled media from being posted
- Skip posting if status changed from 'posting' during the process

### 5. Update Queue Management
**File:** `src/Bot.php` - `addToPostingQueue()` method
- Ensure proper scheduling for album media (group related media together)
- Maintain 1-minute intervals between posts

## Implementation Steps

### Step 1: Database Migration
1. Create new migration file to add required columns
2. Run migration on database
3. Update any existing data if needed

### Step 2: Code Changes
1. Modify `extractMediaInfo()` to detect and extract album information
2. Update `saveMediaToDatabase()` to store new fields
3. Enhance `forwardToBackupChannel()` to include detailed metadata in caption
4. Modify confirmation message to include queue info and cancel button
5. Add cancel callback handling

### Step 3: Testing
1. Test single media upload flow
2. Test album upload flow
3. Test queue calculation and timing
4. Test cancel functionality
5. Verify backup channel forwarding with metadata

## Risk Assessment
- **Database changes:** Low risk, additive only (removed redundant is_album field)
- **Album handling:** Need to ensure proper grouping and scheduling
- **Queue calculation:** Must handle concurrent uploads correctly
- **Backward compatibility:** Existing media records won't have new fields populated

## Dependencies
- Requires database migration before code deployment
- No external API changes needed
- Telegram Bot API already supports media_group_id detection

## Success Criteria
- Media uploads store complete metadata including bot_id, user, album info (via media_group_id), timestamps
- Backup channel receives detailed admin information for each upload
- Users receive queue position, estimated time, and cancel option
- Album uploads are properly detected and grouped (same media_group_id + bot_id)
- Cancellation works for pending media and prevents posting to public channel
- Race condition protection: cancelled media cannot be posted even if cancellation happens during posting process</content>
